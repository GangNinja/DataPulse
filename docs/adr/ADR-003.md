# ADR-003: Use Docker Compose for Local Development

Date: 2026-07-22  
Status: Accepted  
Authors: DataPulse Project Team

## Context

DataPulse uses multiple runtime components. The application needs Python
dependencies, a PostgreSQL database, an ingestion pipeline, and a Streamlit
dashboard. Running all of these directly on a developer machine can work, but
it depends on the correct Python version, installed packages, PostgreSQL setup,
database users, ports, environment variables, and local operating system
configuration.

The project is intended for internship submission and portfolio review, so it
should be easy for another person to run. A reviewer should not need to spend a
large amount of time installing PostgreSQL manually or debugging differences
between local environments.

Docker was introduced to make the runtime environment more consistent and to
show a production-oriented development workflow. Docker Compose was selected
because DataPulse has more than one service and those services need to work
together.

## Problem Statement

The project needs a reliable way to run the database, dashboard, and pipeline
locally. Manual setup creates several possible problems:

- Python dependencies may differ across machines.
- PostgreSQL may not be installed.
- Local database users and passwords may not match the project configuration.
- Ports may conflict with existing local services.
- The dashboard and pipeline need a shared database connection.
- New contributors may not know the exact order to start services.

DataPulse also needs a clean demonstration story. For a portfolio project,
being able to start PostgreSQL and the dashboard with one Docker Compose
command improves reproducibility and makes the project easier to review.

## Decision

Use Docker Compose for local development and portfolio demonstration.

The Compose setup defines a multi-container architecture:

- `postgres`: Runs PostgreSQL and stores data in a named Docker volume.
- `dashboard`: Runs the Streamlit dashboard and connects to PostgreSQL using
  the Docker network.
- `pipeline`: Runs the Python ingestion and transformation pipeline as an
  optional one-off service.

Docker Compose manages service networking, startup order, health checks, port
mapping, volumes, and environment variables. PostgreSQL is exposed to the host
on port `5433` to reduce conflicts with local PostgreSQL installations on
port `5432`. The dashboard is exposed on port `8502` by default so it can run
separately from a local Streamlit process on port `8501`.

The pipeline service uses the same application image as the dashboard but runs
`python main.py` instead of starting Streamlit. This keeps the image reusable
while allowing different commands for different runtime roles.

## Alternatives Considered

### Manual Local Setup

Pros:

- No Docker dependency.
- Easy for developers already comfortable with local PostgreSQL.
- Direct access to local Python and database tools.
- Can be simpler for quick edits and test runs.

Cons:

- Higher onboarding friction.
- More variation between machines.
- PostgreSQL setup can be error-prone for beginners.
- Port, credential, and environment mismatches are common.
- Harder to reproduce the same setup during review.

Manual setup remains supported, but it is not the best default for consistent
project demonstration.

### Single Docker Container

Pros:

- Fewer services to define.
- One container can package Python dependencies.
- Simpler than a full multi-service Compose file.

Cons:

- PostgreSQL and application processes should not be packed into one container.
- Harder to manage database persistence cleanly.
- Does not model realistic service separation.
- Makes it harder to run the pipeline and dashboard independently.

A single container would hide important architecture boundaries. DataPulse has
separate database, pipeline, and dashboard responsibilities, so a multi-service
setup is clearer.

### Full Production Orchestration

Examples include Kubernetes, managed container platforms, or cloud-native job
orchestration.

Pros:

- Better fit for large-scale production deployment.
- Supports scaling, service discovery, rolling updates, and advanced
  operations.
- Can integrate with managed secrets and monitoring.

Cons:

- Too complex for the current project stage.
- Requires cloud infrastructure or extra local tooling.
- Adds operational concepts that distract from the core data engineering goals.
- Not necessary for internship submission or local portfolio review.

Production orchestration may be useful later, but Docker Compose provides the
right balance for the current phase.

## Consequences

Docker Compose becomes the recommended local development and demonstration
runtime for the full system. Developers can still run the Python application
directly with a virtual environment, but Compose provides the most consistent
way to run PostgreSQL, the dashboard, and the pipeline together.

This decision introduces Docker as a project dependency for users who want the
full local stack. It also requires careful documentation so users understand the
difference between local PostgreSQL on port `5432` and Docker PostgreSQL on
port `5433`.

### Positive

- Consistent local environment across machines.
- Easier onboarding for reviewers and contributors.
- Reproducible PostgreSQL setup with a named volume.
- Clear separation between database, pipeline, and dashboard containers.
- Simplified dashboard and pipeline startup.
- Better alignment with deployment-oriented engineering practices.
- Isolation from local Python and PostgreSQL configuration issues.

### Negative

- Requires Docker Desktop or another Docker runtime.
- Containers use disk space and memory.
- Docker networking can be confusing for beginners.
- Local Docker volume data must be managed intentionally.
- Some Windows machines may require Docker Desktop troubleshooting.
- Compose is not a complete production deployment solution by itself.

## Benefits

Docker Compose makes DataPulse easier to run as a complete platform. A reviewer
can start PostgreSQL and Streamlit together, run the pipeline as a separate
service, and inspect the dashboard in a browser. The setup also mirrors a common
real-world pattern where application services connect to a database over a
private network instead of assuming everything runs on the host machine.

For a data engineering portfolio, this demonstrates containerization,
environment consistency, service isolation, and reproducible setup. These are
important professional skills even when the project is not yet deployed to the
cloud.

## Trade-offs

The main trade-off is the requirement for Docker Desktop or another Docker
runtime. Docker Compose improves reproducibility and onboarding, but it adds an
extra tool that users must install and start. This is acceptable because the
project also supports direct local execution with a Python virtual environment
and a separately configured PostgreSQL database.

## Risks

- Docker Desktop may not be installed or running on a reviewer's machine.
- Local port conflicts may still occur if `5433` or `8502` are already in use.
- Accidentally removing Docker volumes will delete local PostgreSQL data.
- Environment variables must be documented clearly to avoid connection errors.
- Compose configuration should avoid hardcoded production secrets.
- The pipeline is still manually triggered and is not scheduled by Compose.

## Future Considerations

- Add a production deployment guide for Render, Streamlit Community Cloud, and
  managed PostgreSQL.
- Add health checks or startup validation for the dashboard service.
- Add a migration step before the pipeline runs if Alembic is introduced.
- Add a scheduled worker or external orchestrator for repeated ingestion.
- Add CI/CD validation for Docker image builds.
- Add separate Compose override files for local development and demonstration.
- Move production secrets to managed secret stores instead of local `.env`
  files.
