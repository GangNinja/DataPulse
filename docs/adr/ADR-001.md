# ADR-001: Core DataPulse Architecture

## Status

Accepted

## Context

DataPulse ingests GitHub repository metadata and prepares it for analytics. The
project needs a foundation that is understandable for a portfolio project while
still matching production data engineering habits.

## Decision

Use PostgreSQL as the warehouse database, SQLAlchemy as the Python ORM, the
GitHub REST API as the source system, and a modular Python architecture split
across configuration, ingestion, database, transformation, utilities, SQL, and
documentation.

## Rationale

PostgreSQL is a strong fit because it supports relational constraints, indexes,
analytical SQL, timestamps, and durable local development without requiring a
larger warehouse platform.

SQLAlchemy gives the project typed ORM models, safer persistence logic, database
session management, and a clean path for adding tables without embedding SQL in
the ingestion flow.

The GitHub REST API is the right initial source because repository metadata is
well documented, useful for analytics, easy to validate, and representative of
real API ingestion work.

A modular architecture keeps responsibilities clear. Ingestion fetches and
validates source data, the database layer owns persistence, transformations
create analytics-ready metrics, SQL files document analysis use cases, and
utilities provide shared behavior such as logging.

## Consequences

The project can now ingest many repositories, protect against duplicate records,
and serve analytics queries from a derived metrics table. Future weeks can add
orchestration, dashboards, dbt, CI/CD, or deployment without rewriting the Week
1 and Week 2 foundation.
