# ADR-001: Use PostgreSQL as the Primary Data Warehouse

Date: 2026-07-22  
Status: Accepted  
Authors: DataPulse Project Team

## Context

DataPulse is a GitHub Repository Intelligence Platform that ingests repository
metadata from the GitHub REST API, stores it for analysis, applies data quality
checks, builds derived metrics, calculates repository health scores, and serves
dashboard and SQL analytics use cases.

The database layer is central to the project. It stores the core repository
records, supports duplicate protection, enforces data constraints, enables
analytics queries, and provides the source tables used by the transformation
and dashboard layers. The project started as a basic ingestion pipeline, but
the target direction is an analytics-ready data platform. That means the
database should support reliable relational modeling, SQL analytics, indexing,
joins, and future scaling without requiring a complete rewrite.

The original architecture decision already identified PostgreSQL, SQLAlchemy,
the GitHub REST API, and modular Python code as the foundation of DataPulse.
This ADR preserves that database direction and documents PostgreSQL as the
primary data warehouse decision in a clearer production format.

## Problem Statement

DataPulse needs a durable storage system for GitHub repository metadata and
derived analytics tables. The storage layer must support:

- Structured repository records with relationships between base and derived
  tables.
- SQL queries for analytics, reporting, and dashboard use cases.
- Constraints that prevent duplicate or invalid records.
- Indexes for common query patterns such as owner, repository name, language,
  stars, forks, and health score.
- Compatibility with SQLAlchemy ORM and Python-based data engineering
  workflows.
- A path toward future deployment, managed databases, and larger data volumes.

A simple local file or document store would be easier at first, but it would
not represent the type of storage system commonly used in production data
engineering projects. DataPulse needs a database that is both practical for
local development and credible for portfolio-level analytical workloads.

## Decision

Use PostgreSQL as the primary data warehouse for DataPulse.

PostgreSQL stores the main `repositories` table, the derived
`repository_metrics` table, and the `repository_health` table. SQLAlchemy ORM
is used by the Python application to define models, create tables, manage
sessions, and insert or update records. SQL files are used separately to
document analytics queries that can be run directly in PostgreSQL tools such as
pgAdmin.

PostgreSQL is selected because it provides strong relational data modeling,
durable storage, mature SQL support, indexing, constraints, transactional
behavior, and compatibility with future deployment options. It also works well
locally through Docker Compose and can later be moved to a managed PostgreSQL
provider such as Neon, Render PostgreSQL, AWS RDS, Azure Database for
PostgreSQL, or Google Cloud SQL.

## Alternatives Considered

### SQLite

Pros:

- Very easy to set up locally.
- No separate database server is required.
- Good for small demos, tutorials, and simple local scripts.
- Works well for lightweight single-user applications.

Cons:

- Less representative of production data warehouse environments.
- Weaker fit for multi-service Docker architecture.
- Limited concurrency compared with PostgreSQL.
- Not ideal for a project that needs warehouse-style tables, dashboard reads,
  and future deployment.
- Fewer options for realistic database administration, roles, network access,
  and managed hosting.

SQLite would have reduced setup effort, but it would make the project feel more
like a local script than a data platform. DataPulse is intended to demonstrate
data engineering practices, so a server-based relational database is a better
fit.

### MySQL

Pros:

- Mature relational database.
- Strong ecosystem and broad hosting support.
- Good performance for many web application workloads.
- Supports indexes, joins, constraints, and SQL querying.

Cons:

- PostgreSQL generally provides richer analytical SQL features and data types.
- PostgreSQL has stronger support for advanced querying, window functions,
  expression indexes, and extensibility.
- PostgreSQL is commonly used in analytics-heavy Python data projects and is a
  natural fit for warehouse-style modeling.
- Some SQL behavior and strictness can vary by MySQL configuration.

MySQL could support the basic project requirements, but PostgreSQL is a better
fit for analytical workloads, data modeling, and future expansion into more
warehouse-like patterns.

### MongoDB

Pros:

- Flexible document model.
- Easy to store raw JSON-like API responses.
- Useful when source data has highly variable structure.
- Good for some application workloads where schema flexibility is important.

Cons:

- DataPulse has strongly relational analytics needs.
- Dashboard and SQL analytics queries benefit from structured tables and joins.
- Constraints and duplicate protection are clearer in a relational database.
- The project needs SQL-first analysis, which is not MongoDB's primary model.
- Derived analytics tables such as metrics and health scores naturally relate
  back to repositories through foreign keys.

MongoDB would be useful if DataPulse mainly stored raw API payloads, but the
project is designed around structured repository intelligence, relational
modeling, and SQL-based analysis.

## Consequences

PostgreSQL becomes the source of truth for repository and analytics data in
DataPulse. The ingestion pipeline writes cleaned and validated records into
PostgreSQL, transformation jobs read from PostgreSQL and write derived tables,
and the dashboard reads analytics-ready data from PostgreSQL.

This decision also means contributors need a PostgreSQL instance available when
running the full application. The project addresses this through Docker Compose
so a local database can be started consistently. For non-Docker development,
contributors can connect to a local or managed PostgreSQL database by setting
`DATABASE_URL`.

### Positive

- Strong relational modeling for repositories, metrics, and health scores.
- SQL-first analytics for pgAdmin, dashboards, and future reporting tools.
- Database constraints help protect against duplicate and invalid data.
- Indexing supports faster reads for common analytical queries.
- Works well with SQLAlchemy ORM and Python testing.
- Easy transition path from local Docker PostgreSQL to managed PostgreSQL.
- More realistic for a data engineering portfolio than a file-based database.

### Negative

- Requires a running database server.
- Local setup is more complex than SQLite.
- Schema changes should eventually be managed with migrations.
- Credentials and connection strings must be handled carefully.
- PostgreSQL-specific SQL syntax may reduce portability to other databases.

## Benefits

PostgreSQL supports the current needs of DataPulse and keeps the project open
for future improvements. The project can perform analytical queries such as top
repositories by stars, language distribution, average health score, stale
repositories, and repositories needing attention. It can also support joins
between source tables and derived tables without flattening everything into one
large record.

For portfolio purposes, PostgreSQL demonstrates practical data engineering
skills: schema design, constraints, indexes, SQL analytics, Dockerized
databases, connection configuration, and integration with Python application
code.

## Trade-offs

The main trade-off is setup complexity. PostgreSQL gives DataPulse a realistic
and scalable warehouse foundation, but it requires a running database,
credentials, ports, and connection management. This is more work than SQLite,
but the trade-off is acceptable because the project is intended to demonstrate
data engineering practices rather than only local scripting.

## Risks

- Database setup problems can block new contributors if documentation is not
  clear.
- Poor indexing choices could slow analytical queries as data volume grows.
- Without migrations, future schema changes may become difficult to manage.
- Local and Docker database connection strings can be confused if environment
  variables are not documented carefully.
- Long-running ingestion could require batching or scheduling improvements as
  the project grows.

## Future Considerations

- Add Alembic for database migrations.
- Add separate staging and analytics schemas if the project grows.
- Add historical snapshot tables for trend analysis.
- Add partitioning if repository history or event data becomes large.
- Add read-only database users for dashboard access.
- Move from local Docker PostgreSQL to managed PostgreSQL for deployment.
- Evaluate columnar or cloud warehouse options if the analytical workload grows
  beyond PostgreSQL's intended role.
