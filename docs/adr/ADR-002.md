# ADR-002: Adopt a Modular ETL Architecture

Date: 2026-07-22  
Status: Accepted  
Authors: DataPulse Project Team

## Context

DataPulse started as a GitHub repository ingestion project and grew into a
larger analytics platform. The project now includes GitHub API ingestion,
database persistence, validation, cleaning, transformation, repository health
scoring, analytics SQL, logging, tests, Docker support, and a Streamlit
dashboard.

As the project expanded, the pipeline needed clear boundaries between the
different responsibilities. Fetching data from GitHub is a different concern
from validating records. Cleaning data is a different concern from writing to
PostgreSQL. Transforming warehouse data into metrics is a different concern
from presenting charts in Streamlit. Keeping these responsibilities separate
makes the project easier to understand, test, maintain, and extend.

The project is also intended to be portfolio-worthy. A modular ETL structure
shows production-oriented thinking and makes it easier for reviewers,
interviewers, and future contributors to understand how the system works.

## Problem Statement

A monolithic script could fetch GitHub data, clean it, write it to PostgreSQL,
calculate metrics, and display results in one place. That approach is quick for
small prototypes, but it becomes difficult to maintain as soon as the project
adds more steps.

DataPulse needs an architecture that supports:

- Clear separation between ingestion, validation, cleaning, transformation,
  analytics, and dashboard layers.
- Unit tests for individual pieces of logic.
- Reuse of database, logging, and transformation utilities.
- Easier debugging when one part of the pipeline fails.
- Future extensions such as orchestration, dbt, CI/CD, scheduling, or cloud
  deployment.
- A professional project structure that communicates how the data flows.

Without modular boundaries, small changes would become risky because one file
would own too many responsibilities. A failure in GitHub ingestion could become
hard to separate from a database issue or dashboard query issue.

## Decision

Adopt a modular ETL architecture for DataPulse.

The project is organized into focused modules:

- `config/` loads environment-driven settings.
- `ingestion/` connects to the GitHub REST API and converts API responses into
  application records.
- `quality/` validates, cleans, checks, and reports on data quality.
- `database/` owns SQLAlchemy models, sessions, and persistence logic.
- `transformations/` builds analytics-ready repository metrics.
- `extensions/` contains the Repository Health Score logic.
- `sql/` stores reusable analytics and health queries.
- `dashboard/` contains the Streamlit dashboard and read-only dashboard data
  access helpers.
- `utils/` contains shared utilities such as logging.
- `tests/` verifies behavior across ingestion, transformations, health,
  database insertion, dashboard helpers, and data quality.

The main pipeline coordinates these modules but does not contain all business
logic itself. Each module owns a focused responsibility and exposes functions
that can be tested or reused independently.

## Alternatives Considered

### Monolithic Pipeline Script

Pros:

- Simple to write in the earliest phase.
- Fewer files for a beginner to navigate.
- Quick for one-off scripts and demos.
- Less initial structure to maintain.

Cons:

- Difficult to test individual behaviors.
- Harder to debug when ingestion, cleaning, persistence, and analytics are
  mixed together.
- Encourages duplicated logic as features grow.
- Makes future dashboard, Docker, and deployment work harder.
- Does not represent production ETL design well.

A monolithic script would have been acceptable for Week 1 only, but it would
not support the larger direction of DataPulse.

### Notebook-Driven Workflow

Pros:

- Good for exploration and visual analysis.
- Easy to inspect intermediate data manually.
- Useful for prototyping formulas and SQL ideas.
- Accessible for learning and demonstrations.

Cons:

- Weak fit for repeatable ingestion and scheduled pipelines.
- Harder to test with pytest.
- Harder to package into Docker containers.
- Can mix exploration, production logic, and output in one place.
- Less suitable for a clean open-source repository.

Notebooks may be useful later for exploratory analysis, but the core ETL
system should remain in Python modules.

### Framework-Heavy Architecture

Examples include Airflow, Dagster, dbt, or a full warehouse modeling framework.

Pros:

- Strong orchestration and transformation patterns.
- Better scheduling and monitoring for production.
- Clear dependency graphs and operational visibility.
- Useful for larger data teams.

Cons:

- Adds complexity before the project needs it.
- Increases setup burden for reviewers and contributors.
- Can distract from the core learning goals.
- May require deployment infrastructure that belongs in a later phase.

Frameworks are valuable future additions, but DataPulse is currently better
served by a lightweight modular Python architecture.

## Consequences

The project has more files than a single-script implementation, but each file
has a clearer purpose. Contributors can make focused changes without reading
the entire application. Tests can target small functions and modules. The
dashboard can read from the database without knowing how GitHub ingestion works.
The data quality layer can be improved without rewriting the database layer.

The modular structure also creates a natural path for future work. For example,
the transformation layer could later move to dbt, the main pipeline could be
scheduled by Airflow or Dagster, and the dashboard could be deployed separately
from ingestion.

### Positive

- Better maintainability because responsibilities are separated.
- Easier unit testing of ingestion, validation, transformations, and scoring.
- Better reuse of shared functions such as logging and database access.
- Easier debugging because failures map to specific layers.
- Better scalability for future features and deployment.
- Clearer portfolio presentation and architecture explanation.

### Negative

- More files and folders for new contributors to understand.
- Requires discipline to keep responsibilities in the right module.
- Some small features may require touching multiple files.
- The pipeline coordinator must clearly document the execution order.

## Benefits

The modular ETL architecture improves DataPulse as both an engineering project
and a learning artifact. Each layer can be discussed clearly:

- Ingestion gets source data.
- Validation checks correctness.
- Cleaning standardizes values.
- Persistence stores durable records.
- Transformation creates analytics-ready outputs.
- Health scoring adds domain-specific insight.
- SQL queries and Streamlit expose the data to users.

This structure makes the project easier to explain in interviews because it
matches how real data platforms are usually discussed: sources, pipelines,
storage, transformations, quality checks, and consumption layers.

## Trade-offs

The main trade-off is that a modular project has more files than a monolithic
script. New contributors must understand where ingestion, validation, cleaning,
database access, transformations, and dashboard logic belong. This structure is
worth the extra navigation because it keeps the project maintainable, testable,
and easier to extend as the platform grows.

## Risks

- Poor boundaries could still cause duplication if modules start importing too
  much from each other.
- Without clear naming, the number of folders may confuse new readers.
- Some logic could drift between dashboard queries and transformation code if
  ownership is not documented.
- As the project grows, a simple pipeline coordinator may become insufficient
  for scheduling, retries, and observability.

## Future Considerations

- Add orchestration with Airflow or Dagster when scheduled runs are needed.
- Add Alembic migrations for database schema evolution.
- Consider dbt for transformation models if SQL transformations become more
  complex.
- Add CI/CD checks so tests and linting run automatically.
- Introduce interface contracts or typed data models if API transformations
  become larger.
- Keep dashboard logic read-only and avoid mixing dashboard code with ETL
  writes.
